# 规划交流指南 (Plan Communication Guidelines)

## 自上而下编排原则 (Top-Down Orchestration)
Agent 在收到任务规划指令时，必须遵循严格的自上而下树状分层构建逻辑：
* **全局上下文优先:** 优先确立 `-workflow.yaml`，界定工作流的宏观目标、成功标准、全局约定与适用边界。
* **主调度框架构建:** 其次构建顶层入口任务 `!entry.task.yaml`，聚焦于任务主要分支的阶段划分、状态流转、异常捕获与宏观路由控制。
* **延迟细节推演:** 严禁在规划初期越级陷入底层的单步命令、具体参数绑定或深层脚本编写。深层子任务 (`*.task.yaml`) 仅在顶层路由框架明确、且确有必要进行逻辑解耦时，方可向下拆解。

## 最小交付与介质规范 (Minimum Deliverable & Media Constraints)
规划的输出必须是机器可读的标准化制品，严禁以中间思考过程替代最终交付：
* **严禁非结构化替代:** 内部推演 (Chain-of-Thought)、条目式大纲、自由文本草稿等仅允许作为 Agent 工作上下文中的临时推演，绝不允许用其替代正式的 YAML 文件输出。
* **交付基线 (Baseline Deliverables):** 任何有效的工作流规划，必须至少交付包含全局上下文的 `-workflow.yaml` 与承载主调度逻辑的 `!entry.task.yaml`。不得以“后续执行时补充”为由省略上述基线交付物。

## 渐进式演进与状态收口 (Progressive Evolution & Stage Closure)
规划阶段是一个允许迭代的演进状态，而非一次性的绝对定稿：
* **入口任务的暂定性 (Provisional Entry):** 承认 `!entry.task.yaml` 在早期的草案属性。随着执行路径的探明、环境约束的暴露或用户补充修正，顶层路由结构允许在后续交互中被重构。
* **准出标准 (Closure Criteria):** 规划阶段转入正式执行阶段需满足以下收口条件：
    1. 已生成符合基线规范的最小交付物。
    2. 顶层目标与主分支的基本路由流转关系已明确。
    3. 缺乏环境先验的底层参数与模糊逻辑已显式标记为 `unknown`，严禁基于猜想进行填补或伪造。

## 规划期的交互与人机对齐 (Interaction & Alignment During Planning)
规划过程严禁陷入 Agent 单向的静默建模。必须将用户作为规划闭环中的核心决策与审查节点，规范交互时机与汇报结构：
* **主动挂起与请求介入 (Active Intervention Triggers):** 在规划推进过程中，若遭遇以下高价值或高不确定性节点，Agent 必须立即挂起规划生成，向用户抛出选项或请求确认，严禁盲目发散：
    - **策略分歧:** 顶层目标的实现存在多个成本、风险或时间周期差异显著的并行策略方向。
    - **关键约束缺失:** 缺失足以直接影响全局模块划分或路由走向的核心前置依赖（如缺乏目标集群的鉴权方式或底层数据的物理位置）。
    - **主观偏好依赖:** 路由结构直接受制于用户的业务偏好。
* **结构化阶段汇报 (Structured Milestone Reporting):** 初始基线 YAML 产出后，Agent 严禁仅输出文件内容后静默结束。必须向用户提供自然语言的规划摘要，以供审阅与修订。该汇报必须包含：
    - **核心流转提要:** 当前工作流的总体目标与主要执行阶段划分。
    - **不确定性暴露:** 明确列举当前被标记为 `unknown` 的重要信息、关键假设以及潜在风险。
    - **断点预告:** 提示工作流中已主动植入的人机审核断点 (Breakpoints) 位置及预期审查内容。
    - **下一步引导:** 明确向用户抛出待确认事项，请求用户批准转入执行态、提供修正意见或补充缺失的子任务信息。
